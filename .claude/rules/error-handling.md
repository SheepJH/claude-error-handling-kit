# 에러 처리 규칙

이 프로젝트에서 코드를 작성하거나 수정할 때 아래 규칙을 반드시 지킨다.

## 원본 에러 보존

- catch 블록에서 에러를 일반적인 메시지로 덮어쓰지 않는다. 원본 에러를 반드시 보존한다:

  ```ts
  // ❌ 금지 — 원인이 사라짐
  catch (err) {
    throw new Error("생성에 실패했습니다");
  }

  // ✅ 필수 — 원본 에러를 cause로 연결
  catch (err) {
    throw new Error("사용자 생성 실패", { cause: err });
  }
  ```

- 빈 catch 블록 금지. 의도적으로 무시하는 경우 그 이유를 주석으로 남긴다.
- 에러를 다시 던지지 않고 그 자리에서 처리하는 경우에도 `console.error`(또는 로거)에 원본 에러 전체를 남긴다.

## API / 외부 시스템 호출

- HTTP 요청 실패 시 에러 메시지에 반드시 포함할 것: **HTTP status, 요청 URL(메서드 포함), 응답 body**.
  "Failed to fetch"나 "요청 실패(400)" 같은 메시지만 남기는 것은 금지.
- 외부 시스템 경계(DB, 외부 API, AI API, 파일 IO, 캐시 등)마다 어느 레이어에서 실패했는지
  식별 가능한 prefix를 에러 메시지에 붙인다. 예: `[DB]`, `[Anthropic]`, `[S3]`, `[Redis]`
- 외부 API의 에러 응답 body에는 실패 원인이 들어있는 경우가 많다. 상태 코드만 보고 버리지 말고
  body를 파싱해서 에러 메시지에 포함시킨다.

## 에러 인프라 (프로젝트 초기에 먼저 구축)

- 모든 HTTP 호출은 공용 fetch 래퍼(예: `apiClient`)를 통과시킨다. 래퍼는 실패 시
  status / URL / 응답 body를 담은 구조화된 에러를 던진다. 개별 호출부에서 fetch를 직접 쓰지 않는다.
- 서버에는 에러 미들웨어를 하나 두고, 모든 에러를 요청 ID와 함께 스택 포함 로깅한다.
- 새 외부 연동(DB, AI API 등)을 추가할 때는 해당 연동의 에러 변환 레이어를 함께 만든다.

## 사용자에게 보여주는 에러

- UI에는 사용자용 메시지를 보여주되, 원본 에러(status, body, 스택)는 콘솔/로그에 반드시 남긴다.
- 개발 모드에서는 UI에도 상세 에러(status, 응답 body)를 노출해서 디버깅 가능하게 한다.
- toast나 alert에 하드코딩된 일반 메시지("오류가 발생했습니다")만 띄우고 끝내지 않는다.
  최소한 어떤 작업이 실패했는지 + 서버가 준 메시지를 표시한다.

## 검증

- 기능 구현이 끝나면 에러 경로도 동작하는지 확인한다: 의도적으로 잘못된 입력을 넣거나
  외부 서비스가 실패하는 상황을 가정했을 때, 로그와 UI에서 실패 지점을 특정할 수 있어야 한다.
